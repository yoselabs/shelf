# PDF engine comparison — findings (2026-07-09)

Ran per `run_bench.py` against the corpus in `bench/corpus/` (see its README for
sourcing/citations). `docling` and `markitdown[pdf]` are D4-isolated probes for
their PDF paths specifically (markitdown's base install in `convert-md[documents]`
never enabled `[pdf]`); `pymupdf4llm` and `unstructured[pdf]` are fully isolated,
transient, never added to `pyproject.toml`. `marker-pdf` probed separately (heavy
CPU-only run) — see the marker addendum below/forthcoming.

## Engine notes

- **docling** — required a platform-specific fix: its default device selection hit
  `Cannot convert a MPS Tensor to float64` on Apple Silicon (docling/torch's MPS
  backend doesn't support float64 for an op it uses). Forced `AcceleratorDevice.CPU`
  in `bench/engines/run_docling.py`; all 4 cells then passed. Real, reproducible
  platform bug, not a fidelity finding — but worth knowing if this ever runs in
  convert-md's actual runtime on Apple Silicon without the same workaround.
- **markitdown** — base `convert-md[documents]` install doesn't include `markitdown[pdf]`;
  it was only ever wired for pptx/xlsx (see `dispatch.py`). Probed with the extra
  installed isolated, so the comparison is fair to markitdown's real PDF capability.

## Scoring — D2 axes

| Axis | docling | pymupdf4llm | markitdown | unstructured |
|---|---|---|---|---|
| **Table structure** (table_heavy.pdf) | Clean pipe tables, correct rows/cols across multi-row-header IRS tables | Clean pipe tables, correct structure (adds bold-cell markup, verbose but structurally sound) | **Fabricates false tables** — 2-column-layout text gets misread as table cells, producing garbage pipe-syntax with meaningless splits (`table_heavy.md:830-863`) | No table markdown emitted at all (runner limitation — joins plain text without checking `element.metadata.text_as_html`; see caveat below) |
| **Heading hierarchy** (multicolumn_academic.pdf, prose_report.pdf) | Recovered correctly, flat `##` (no nesting) | Recovered correctly, **correctly nests** `#`/`##`/`###` by section depth | **Zero headings recovered** on either file | Recovers real headings **plus ~150+ false positives** — chart axis labels, figure fragments tagged as `##` (`unstructured__prose_report.md`, e.g. `## 6.0Bits per word`, `## Prediction`) |
| **Reading order** (table_heavy.pdf's actual 2-column IRS layout, multicolumn_academic.pdf) | Correct — paragraphs flow in true reading order | Correct | **Scrambled** — columns interleaved mid-sentence (`markitdown__table_heavy.md:6-9`: "General Instructions Part II ... No Tax on Tips ... You have a valid social security" jams two columns together) | **Scrambled**, plus stray single-word "headings" from column-break artifacts (`unstructured__scanned_ocr.md`: `## the`, `## of`, `## Among the`) |
| **OCR/scanned-page handling** (scanned_ocr.pdf) | Clean (but see caveat below) | Clean | Clean but preserves mid-word hyphenation breaks (`Work-\nmen`) | Clean text, but same reading-order fragmentation as above |
| **Noise** (headers/footers/page artifacts) | Minimal | Minimal | Retains page-furniture lines (`Page 101 of 126 15:15 - 25-Feb-2026`, `Fileid: …`) verbatim mixed into body | Retains same page-furniture lines, plus the false-heading noise above |

**Caveat — scanned_ocr.pdf didn't actually test OCR.** The archive.org-sourced
scan (`leavesofgrass00whit`) ships with its own pre-baked OCR text layer, so
every engine just extracted embedded text rather than running its own
OCR/vision path. This axis is **not validated** by this run — a true
image-only PDF (no text layer) would be needed to test it, and none of the
four public-domain sources tried had one conveniently available. Flagging
as a known gap in the corpus, not a finding about the engines.

**Caveat — unstructured's table score is an artifact of the runner, not
necessarily the engine.** `run_unstructured.py` joins `element.text` only;
`unstructured` actually classifies `Table` elements and exposes
`element.metadata.text_as_html`, which was not used. Re-running with that
would likely change unstructured's table-structure score — noted for anyone
re-running this bench, not fixed here (D4: this bench is re-runnable,
findings are dated evidence not a one-time verdict).

## Provisional read (pre-marker)

**docling and pymupdf4llm are both strong; markitdown and unstructured are
both meaningfully worse on this corpus**, specifically on reading order and
heading/table structure — the two axes that matter most for turning a PDF
into something an LLM can reason over without re-deriving document structure
itself.

Per D3: pymupdf4llm is conflict-free and a2web-installable; docling is not.
**pymupdf4llm is the candidate that actually resolves a2web's blocked case**
— docling remains the fidelity ceiling for consumers that can absorb the tax,
but a2web specifically needs pymupdf4llm (or something that clears its bar)
to have a PDF tier at all.

## Marker addendum (2026-07-09, CPU-only)

Ran separately (`uv run --with marker-pdf`, isolated per D4) — 3/4 cells
completed, 1 crashed:

| PDF | Status | Elapsed |
|---|---|---|
| multicolumn_academic.pdf | ok | 586.0s |
| prose_report.pdf | **error** — `FileNotFoundError` from `pypdfium2` deep inside marker's table processor, on an absolute path that exists; looks like a transient race/resource issue under this sandbox's memory pressure, not a real missing-file bug | 134.6s |
| scanned_ocr.pdf | ok | 876.6s |
| table_heavy.pdf | ok | 492.2s |

**Quality (on the 3 that succeeded):** on par with docling/pymupdf4llm —
correct reading order, correctly-nested headings (`### 3.1` under `## 3.`,
matching pymupdf4llm's nesting, better than docling's flat `##`), comparable
table structure (32 pipe-table lines on table_heavy.pdf, same as
docling/pymupdf4llm), extracts image placeholders. One minor OCR-shaped typo
observed ("vou" for "you") not seen in docling/pymupdf4llm's output.

**Speed — confirms design.md D2's own prediction, worse than predicted:**
492-877 seconds per document (excluding the crash) vs. 13-56 seconds for
docling and 3-13 seconds for pymupdf4llm on the same corpus. That's roughly
**15-60x slower**, not the "5-10x" design.md estimated — CPU-only marker is
categorically impractical for anything but an offline, patience-optional
batch job in this environment.

**Verdict: marker does not change the decision.** Its fidelity is
competitive, but between the GPL-3.0 license (an unresolved adoption
question per the Open Questions section), the GPU-preferred architecture
that doesn't fit a serverless/CLI fetcher shape, and a >30x-worse-than-modeled
CPU runtime, it stays exactly where D2 scoped it: stretch/last-resort, not a
default. pymupdf4llm remains the engine that actually resolves a2web's
blocked case at a cost profile anyone can accept.

## Extended corpus addendum (2026-07-09): CJK, math, and infographic axes

The corpus was extended with 4 files the original 4-category set never
covered (see `bench/corpus/README.md` for full citations): `ja_service_manual.pdf`
(Japanese, Brother sewing machine manual), `zh_service_manual.pdf` (Simplified
Chinese, Brother printer manual), `math_heavy.pdf` (arXiv 2502.17533, dense
LaTeX-rendered formulas), `investor_infographics.pdf` (Toyota Integrated
Report 2025, infographic-dense layout). Ran all 5 engines (including marker
this time — all 20 cells completed `ok`, no crashes) via `run_bench.py`'s new
file filter. This run is investigative: checking whether these axes change the
pymupdf4llm-primary/docling-fallback/marker-stretch-only decision, not
re-litigating the original corpus's result.

### CJK text integrity (ja_service_manual.pdf, zh_service_manual.pdf)

Two real, reproducible defects found in the current production chain:

- **pymupdf4llm fragments vertical Japanese sidebar labels into one
  character per line.** The sidebar heading "下糸をセットしましょう" ("Let's
  set the bobbin thread") comes out as `下 糸 を セ ッ ト し ま し ょ う` across
  10 separate lines (`pymupdf4llm__ja_service_manual.md:213`) — unreadable as
  a heading, and structurally indistinguishable from body text to a
  downstream reader. `markitdown` has the same defect, worse (bold-wrapped
  single characters, e.g. `**ぬ** **う** **前** **の** **準** **備**` for "ぬう前の準備",
  `markitdown__ja_service_manual.md:25-30`). `docling` and `marker` do not
  exhibit this — both render the same vertical sidebar text as a normal
  heading line (`docling__ja_service_manual.md:19`: `## お願い` /
  coherent headings throughout) or skip it via image placeholders.
- **pymupdf4llm and markitdown both leak raw font-glyph-IDs / mis-decoded
  icon-font characters into extracted text on zh_service_manual.pdf.**
  A custom icon font (control-panel button glyphs) has no usable
  ToUnicode mapping. pymupdf4llm renders it as garbage codepoints from
  unrelated Unicode blocks — `᳈໮`, `໡ॄ`, `ᅝܼ  ড়` (Devanagari/Combining-Diacritical
  ranges, `pymupdf4llm__zh_service_manual.md:115-118`) — injected directly
  into a table cell next to real Chinese text. `markitdown` is worse: it
  emits literal unresolved glyph-ID tokens as text, e.g.
  `(cid:7368)(cid:3822)`, `(cid:38)(cid:55)(cid:47)(cid:47)(cid:59)`
  (`markitdown__zh_service_manual.md:11-17`) — tokens that read as
  plausible-looking noise an LLM consumer could easily mistake for real
  (if foreign) content. `docling` and `marker` both sidestep this
  entirely by emitting an image placeholder for the icon instead of
  attempting to decode it — zero garbage text in either output.
- **marker has its own, different CJK defect**: on `ja_service_manual.pdf`,
  several icon/checkbox-adjacent content blocks come out as literal
  repeated-`a` garbage (`aaaaaaaaaaaaaaa`, `marker__ja_service_manual.md:77,89,122`)
  instead of the real (or omitted) content — a distinct failure mode from
  the glyph-ID leak above, but still wrong output, not just missing output.
- `docling` was the cleanest engine on this axis for both files: no
  glyph-ID leakage, no vertical-text fragmentation, coherent Japanese and
  Chinese headings and body text throughout.

**CJK verdict:** real, citable defects in pymupdf4llm (the current primary)
on both scripts. Neither is fatal — most body text stays readable, and the
defects are localized to sidebar UI labels and icon-adjacent table cells,
not the instructional prose that carries the actual content — but this is a
genuine gap, not a clean pass. Flagging as a known limitation rather than
grounds to re-open D3 on its own: the original 4-category corpus was
CJK-free, so this defect was invisible until now and should be tracked, not
buried.

### Math fidelity (math_heavy.pdf)

This axis produced the sharpest finding of this addendum:

- **docling drops every formula in the document.** All display and matrix
  equations come out as the literal placeholder `<!-- formula-not-decoded -->`
  (`docling__math_heavy.md:29,33,39,47,51,59,65,116` — 8 occurrences, i.e.
  every formula in the first 10 pages). On a math-heavy paper, this is a
  total failure of the axis for the current production *fallback* engine —
  not a fidelity tax, a complete content loss.
- **pymupdf4llm partially fails.** Inline math survives as flattened
  plain-text approximations (`un = a1,nun−1 + ...`, readable but stripped of
  proper subscript/superscript structure), but multi-line display equations
  and matrices degrade to blank or malformed layout — the companion matrix
  `CM(n)` renders with its bracket structure but no visible entries
  (`pymupdf4llm__math_heavy.md:71-90`), a data loss a downstream reader
  cannot recover from.
- **markitdown fails catastrophically, differently.** Flowing prose loses
  all inter-word spacing — `Theconstantπhasfascinatedscholarsthroughoutthecenturies`
  (`markitdown__math_heavy.md:11`) — and ordinary paragraphs get randomly
  fragmented into garbage pipe-tables (`markitdown__math_heavy.md:12-38`,
  e.g. a table row containing only `formulasforitsevaluation...Despite`).
  Both prose and math are destroyed on this file.
- **unstructured also fails**: rotated arXiv margin-stamp text (the vertical
  "arXiv:2502.17533v4 [math.OH] 17 Mar 2026" stamp) gets extracted
  character-by-character out of order and interleaved with real body text
  as noise lines (`unstructured__math_heavy.md:1-13`: `6 2 0 2`, `r a`,
  `M 7 1`, `O H . h t a m`) — a distinct reading-order defect from the
  original corpus's column-scrambling, but the same root cause (rotated/
  non-linear source content).
- **marker is the only engine that renders correct, readable math.** Proper
  LaTeX throughout: `$$CM(n) := \begin{pmatrix} 0 & 0 & \dots ... \end{pmatrix}$$`
  with real matrix entries (`marker__math_heavy.md:21`), correct inline math
  (`$\pi$`, `$\delta$`), and a correctly-rendered results table with LaTeX
  fractions inside cells (`marker__math_heavy.md:99-104`,
  e.g. `$\frac{\pi}{2}$`). Clean, faithful, no losses observed.

**Math verdict: this DOES challenge the tidy read of the existing decision.**
Both engines in the current production chain — pymupdf4llm (primary) and
docling (fallback) — have real, severe defects on math-heavy content:
docling's is total (zero formulas survive), pymupdf4llm's is partial but
still a genuine data loss on display equations. marker, the engine already
scoped as "stretch/last-resort" purely on cost grounds, is unambiguously the
best performer on this specific axis. This does **not** by itself justify
promoting marker to primary — its GPL-3.0 license question and >30x-worse
CPU runtime (marker addendum, above) are unchanged and still disqualifying
for a default path — but it means **math-heavy PDF handling is an open gap
in convert-md's PDF chain, not a solved problem**. Any consumer whose
corpus includes scientific papers, engineering specs, or formula-bearing
financial documents should not assume today's default chain preserves that
content; it currently does not. Flagging for a future decision (a
math-aware primary engine, a marker-for-detected-math-documents routing
rule, or an accepted known limitation) rather than resolving it here — out
of scope for this addendum per its directive.

### Infographic / mixed-layout handling (investor_infographics.pdf)

**Re-trimmed 2026-07-09** to pages 162-166 (was 12-23, which landed on
CEO-message prose — see corpus/README.md). The new range covers a "History"
timeline infographic, a "Domestic/Overseas Vehicle Production" chart, and a
real "Financial Summary (Consolidated)" multi-year data table — genuinely
chart/table-dense content. Re-ran all 5 engines. This addresses a direct
follow-up question: **when graphics retention doesn't matter and only
text/table content retention does, which engine wins?**

- **The Financial Summary table's *data* survived intact in every engine** —
  all 11 fiscal years' worth of figures (e.g. Sales Revenues `28,403.1` →
  `48,036.7`, Total Assets `47,427.5` → `93,601.3`) are present and correct
  in `docling`, `pymupdf4llm`, `markitdown`, `unstructured`, and `marker`
  alike. The differentiator is entirely **table structure**, not data loss:
  - **`marker` renders the table perfectly**: a single, correctly-shaped
    markdown table, one row per metric, one column per fiscal year, no
    duplication (`marker__investor_infographics.md:39-61`). The cleanest
    structural result of any engine on this corpus, full stop.
  - **`markitdown` renders it as clean flowing text**, not a markdown table,
    but with each metric on one line in correct left-to-right year order —
    e.g. `Consolidated vehicle sales (Thousands of units) 8,681 8,971 8,964
    ... 9,443 9,362` (`markitdown__investor_infographics.md:124`). Not
    structured markup, but fully linear and trivially re-parseable — a
    strong second place for a text-only consumer.
  - **`docling` and `pymupdf4llm` both produce structurally broken tables**:
    the header row repeats the literal string "Fiscal years ended March 31"
    across all 14 columns (`docling__investor_infographics.md:93-94`), and
    several data rows collapse every column's value into a single jammed
    cell separated only by spaces (`docling__investor_infographics.md:98,
    106-111`; same pattern in `pymupdf4llm__investor_infographics.md:304,
    310-312`). The values are all there and correct, but a naive
    row/column reader would misparse these rows.
  - **`unstructured` destroys the table structure entirely**: every cell
    becomes its own standalone line with zero row/column association — the
    2025 column's `9,362` / `153` / `48,036.7` values are even split off
    into a disconnected block roughly 450 lines later in the file
    (`unstructured__investor_infographics.md:387-406` vs. `859-895`).
    Reconstructing the table from this output requires positional
    inference across nearly 900 lines; for a text/table-content-priority
    use case this is effectively data loss, even though every individual
    number is technically present somewhere in the file.
- **The History timeline's text (model names + years) is a second, separate
  differentiator** — and it inverts the ranking above:
  - `pymupdf4llm`, `markitdown`, and `unstructured` all extract the timeline
    labels correctly attributed to their years — e.g. "Launch of Toyoda
    Model AA passenger car (1936)" appears intact in all three
    (`pymupdf4llm__investor_infographics.md:192`,
    `markitdown__investor_infographics.md:70`,
    `unstructured__investor_infographics.md:131,235`).
  - **`docling` and `marker` both drop the entire History timeline**,
    replacing it with a bare `<!-- image -->` /
    `![](_page_1_Figure_7.jpeg)` placeholder
    (`docling__investor_infographics.md:89`,
    `marker__investor_infographics.md:33`) — real content loss, not a
    graphics-only omission, since the timeline is fundamentally text (model
    names, years, event labels) laid out graphically, not a photo.
- Chart *data values* (the production-volume/financial bar charts on page
  165) are only partially recoverable as text, and only from `pymupdf4llm`
  and `markitdown` — both dump the chart's axis tick labels and per-bar
  values as loose numbers with no attached labels
  (`pymupdf4llm__investor_infographics.md:431-708`,
  `markitdown__investor_infographics.md:169-227`) — technically present,
  practically unusable without the visual to correlate them. `docling`,
  `unstructured`, and `marker` don't surface these numbers as text at all.

**Infographic verdict — direct answer to "which engine retains text/table
content best, graphics aside":** there is no single winner; it depends on
which sub-content matters more. **`marker` wins outright on the actual
financial data table** (correct structure, zero cleanup needed) but **loses
the timeline text entirely**. **`markitdown` is the best all-rounder**: a
clean (if non-tabular) rendering of the financial data in correct order,
plus the History timeline text intact — the only engine that keeps both.
`pymupdf4llm` (today's primary) also keeps both, with correct data
throughout, but wrapped in the same duplicated-header table corruption as
docling and buried in a large volume of orphaned chart-axis-number noise.
`unstructured` is the clear loser for structured/tabular use — data present,
structure destroyed. `docling` sits in the middle: correct but malformed
table, and — like marker — silent content loss on the timeline. **This does
not change the production decision** (marker's cost/license issues and
pymupdf4llm's conflict-free status are unaffected), but it sharpens the
known-limitations list: pymupdf4llm's table *structure* (not its data) is
unreliable on multi-header financial tables, a distinct defect from the math
and CJK findings above.

### Net read

CJK and infographic findings are real but non-disqualifying gaps in the
current chain — worth tracking, not worth reopening D3. The re-trimmed
infographic run sharpens the infographic gap specifically: pymupdf4llm
(today's primary) never loses data, but its financial-table *structure*
breaks on multi-header tables (duplicated header cells, jammed-together
rows) in a way markitdown's plainer linear rendering avoids — a distinct,
newly-identified defect from the CJK glyph-leak and math-loss findings, not
a restatement of them. **Math is the one finding here that should not be
quietly absorbed**: the current pymupdf4llm-primary/docling-fallback chain
loses mathematical content (partially or totally) on formula-bearing PDFs,
and the only engine evaluated that handles math correctly (marker) remains
excluded on cost/license grounds unrelated to fidelity. Both the math and
table-structure findings are open items, not decided ones — recorded here
for whoever picks it up next.
