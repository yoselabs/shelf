# docx engine comparison — findings (2026-07-10)

Companion to `2026-07-09-findings.md` (PDF). Question: `convert-md`'s `.docx`
chain was `[pandoc, markitdown]`, inherited from R142 on one unverified claim —
pandoc's `--track-changes=all` preserves redlines. This bench tests that claim
and the alternatives on a synthesized per-capability corpus. Human-judged,
model-free (same discipline as the PDF findings).

## Corpus (design.md D2 — synthesize, one capability per fixture)

`bench/corpus/docx/`, generated by `_generate.py` (python-docx, run isolated):
`clean_baseline` (headings + emphasis), `table_nested`, `list_nested`,
`footnote`, `tracked_changes` (real `w:ins`/`w:del` markup), `image_embedded`.
Not the user's own documents (privacy boundary carried from the PDF corpus).

## Engines (design.md D3 — pandoc scored in both track-changes modes)

`pandoc-accept`, `pandoc-all`, `markitdown`, `mammoth` (mammoth → html2text).
All 24 cells ran clean (isolated `uv run --with`, D4).

## Scored grid

| Capability | pandoc-accept | pandoc-all | markitdown | mammoth |
|---|---|---|---|---|
| headings + emphasis | ✓ | ✓ | ✓ | ✓ |
| table | ✓ correct header | ✓ | ✗ blank header, real header → body row | ✓ correct header |
| nested list | ~ flat + `<!-- -->` noise | ~ same | ✓ clean (flat) | ~ flat + blank-line noise |
| footnote | ✓ true `[^1]` | ✓ | ✓ link-style + backref | ✓ `[1]` + note |
| tracked changes | ✓ clean final text | ✗✗ HTML-span redline soup | ✓ clean | ✓ clean |
| embedded image | ✗ dangling `media/` ref | ✗ same | ✓ base64 inline | ✓ base64 inline |
| weight | ~100MB+ binary | same | 23 pkgs (already a dep) | featherweight (`cobble` only) |

## Decisive findings

1. **pandoc's sole differentiator is a live latent bug, not a feature.**
   `--track-changes=all` — *exactly what convert-md shipped* through v0.6.0 —
   rendered the tracked-changes fixture as
   `The migration took <span class="deletion"...>four</span><span class="insertion"...>two</span> weeks...`
   i.e. HTML-span redline soup, with both deleted and inserted text jammed
   together, injected into the vault. In `accept` mode pandoc is merely *equal*
   to the light engines. So pandoc offers nothing they don't, except the
   ability to emit content nobody wants. **Dropped.**

2. **pandoc fumbles embedded images** — a dangling `media/image1.png` reference
   it never extracts (no `--extract-media`). mammoth/markitdown self-contain as
   base64 data URIs (valid, vault-portable). Reinforces the drop.

3. **The table split is markdownify vs html2text, not markitdown vs mammoth.**
   markitdown wraps mammoth internally, so both consume the *same* header-less
   HTML mammoth emits (`<tr><td>…`, no `<thead>`/`<th>`, because python-docx
   didn't mark a header row). The divergence is entirely in the HTML→markdown
   stage: markitdown ships **markdownify** (emits an empty header row, demotes
   the real header to a body row — conservative, faithful, ugly); the mammoth
   path uses **html2text** (promotes row 1 to header — a heuristic, friendlier).
   Data survives in both. Only bites *unmarked-header* tables; a marked header
   (`<th>`) renders correctly in both.

4. **Never route docx through the generic web pipeline.** Feeding mammoth's HTML
   to `convert_html()` (trafilatura-primary) was *much worse*: trafilatura is a
   web-page boilerplate *remover*, and on a clean fragment it stripped all
   headings and emphasis and exploded the table (one cell per line) — while
   silently grading `high`. trafilatura is an extraction pre-stage web pages
   need; docx-derived HTML is already clean and must go straight to the render
   stage (html2text). Verified separately that trafilatura on a *real full page*
   (with boilerplate + a marked-header table) keeps every heading/table and
   correctly strips nav/footer — so the web path (a2web) is healthy; the two
   paths must simply never cross. This motivated the `source_kind` API (D7).

## Decision

- **Drop pandoc entirely.** Remove `pypandoc-binary` from `convert-md[documents]`
  (a ~100MB+ binary carried for negative marginal value).
- **`.docx` → `[MammothEngine, MarkitdownEngine]`.** Mammoth renders clean HTML
  and routes through convert-md's own `source_kind="clean"` path (html2text) —
  one HTML→markdown renderer shared with the web path, not a third opinion.
  markitdown stays as the cheap already-a-dep fallback if mammoth fails.
- **`convert-md-v0.7.0`** (behavior-changing engine removal → minor bump).
  Recorded in ledger.

## Caveats / open axes (not blockers)

- **Unmarked-header table only:** finding 3's divergence vanishes on marked-header
  tables. A marked-header fixture would tighten it but doesn't change the
  decision (mammoth is the more robust default either way).
- **OMML math untested:** synthesis-weak (D2 exception); no fixture yet. Likely
  rare in a2kay's KB diet; flagged as a follow-up axis, same as the PDF bench
  left math an open gap.
- **Nested-list depth flattened by all engines:** but the fixture used named
  styles (`List Bullet 2`), which may not encode true `numPr`/`ilvl` nesting —
  a fixture weakness, not necessarily an engine failure. Re-test with real
  numbering-level nesting if list depth ever matters.
